# Design: Terminology Documentation & Subagent Tracking

**Date:** 2026-07-20  
**Status:** Approved  
**Scope:** Two independent improvements to `@alex123bob/opencode-cache-stats`

---

## 1. Background

The plugin currently tracks 6 cumulative session metrics (Hit Rate, Read, Written, Raw Input, Output, Turns) and displays them in a TUI sidebar widget. Two gaps were identified:

1. **Terminology confusion** — users may not know what each metric means, and it's unclear that all sidebar values are cumulative across the session rather than per-turn.
2. **No subagent awareness** — subagent turns are silently folded into the same session stats as the main agent with no distinction, making it impossible to see how much work each agent did.

---

## 2. Feature 1: Terminology Documentation

### Approach

- Add a **"Understanding the metrics"** section to `README.md`
- Lead with an explicit callout: all sidebar values are **cumulative for the session**; the JSONL file records **per-turn** values
- Include a definitions table for all 6 terms
- No TUI layout changes (sidebar space is too constrained for a legend)

### README section (to insert after the sidebar example block)

```markdown
## Understanding the metrics

> **All sidebar values are cumulative for the entire session** — they accumulate
> across every turn, not reset per turn. The JSONL file records per-turn values
> so you can analyse individual turns separately.

| Metric | What it means |
|---|---|
| **Hit rate** | `cacheRead / (cacheRead + cacheWrite + rawInput) × 100` — the fraction of total input tokens served from cache. Higher is cheaper and faster. |
| **Read** | Tokens retrieved from the prompt cache (cache hits). These are not re-processed by the model, so they cost less. |
| **Written** | Tokens written into the prompt cache this session (cache misses that seeded the cache). Only shown when non-zero. Not all providers report this. |
| **Raw input** | Fresh, uncached input tokens — the part of your prompt that was processed from scratch. |
| **Output** | Completion tokens generated by the model. |
| **Turns** | Number of completed assistant responses in this session. |
```

### JSONL `hitRate` note

The `hitRate` field in the JSONL file is **per-turn** (computed from that turn's tokens only), while the sidebar hit rate is **session-cumulative**. The README should note this distinction.

## 3. Feature 2: Subagent Tracking

### Approach

Use `parentSessionID` from the event payload to build an agent tree. Each unique `sessionID` is its own `AgentEntry`. Active agents float to the top of the sidebar; finished agents are dimmed and sink below. Degrades gracefully if `parentSessionID` is absent (treated as main agent).

### 3.1 Data Model (`shared.ts`)

#### New type: `AgentEntry`

Replaces the raw `SessionStats` map value:

```typescript
type AgentEntry = {
  sessionID:  string
  parentID:   string | null   // null = main agent; populated from parentSessionID
  label:      string          // "Main Agent" | "Subagent 1" | "Subagent 2" ...
  isActive:   boolean         // true while turns are still arriving
  lastActive: number          // Date.now() at last completed turn — used for sort
  stats:      SessionStats
}
```

`SessionStats` itself is **unchanged**.

#### Updated type: `JsonlRecord`

Two new fields appended (non-breaking — old consumers ignore unknown fields):

```typescript
type JsonlRecord = {
  // ... all existing fields unchanged ...
  parentSessionID: string | null   // null for main agent turns
  agentLabel:      string          // "Main Agent" | "Subagent N"
}
```

### 3.2 Agent identity rules

On first `message.updated` event for a `sessionID`:

| Condition | Label assigned |
|---|---|
| `info.parentSessionID` is absent or null | `"Main Agent"` |
| `info.parentSessionID` is present | `"Subagent N"` where N = spawn order (1-indexed) |

Spawn order is determined by order of first appearance across all events, not by any server-assigned index.

### 3.3 Active / finished detection

- `isActive = true` when a new turn arrives for a session
- `isActive = false` after **30 seconds of idle** — no new turns for that session
- Implemented via a per-session `setTimeout` reset on each turn; cleared on plugin dispose
- If a future opencode API exposes an explicit session-end event, that takes priority over the timeout

### 3.4 Sidebar rendering

#### Sort order

```
1. Active agents   — sorted by lastActive descending (most recently active first)
2. Finished agents — sorted by lastActive descending (most recently finished first)
```

Main agent has no special position; it floats or sinks purely by activity.

#### Section format

**Active agent:**
```
── Main Agent (active) ───────
  Hit rate:  68.9%
  Read:      1,240 tok
  Written:     512 tok
  Raw input:   308 tok
  Output:      320 tok
  Turns:           3
```

**Finished agent (dimmed via `textMuted` color):**
```
── Subagent 1 ────────────────
  Hit rate:  42.1%
  Read:        890 tok
  Raw input:   210 tok
  Output:      180 tok
  Turns:           5
```

- `(active)` label appended to header only while `isActive = true`
- Finished sections rendered in `textMuted` fg color
- Each section separated by a blank line

#### New render function signature

`renderCacheStats(stats: SessionStats | undefined)` is **removed** and replaced entirely by `renderAllAgents(agents: AgentEntry[])`. There are no other callers of `renderCacheStats` outside `tui.ts`.

Returns empty string if `agents` is empty (widget hidden until first turn).

### 3.5 TUI widget changes (`tui.ts`)

- `sessionStats: Map<string, SessionStats>` → `agents: Map<string, AgentEntry>`
- `subagentCount: number` tracks how many subagents have been registered (for labelling)
- `CacheStatsWidget` no longer scopes to a single `sessionID` from `slotProps` — renders the full sorted agent list
- Per-session idle timers managed in a `Map<string, ReturnType<typeof setTimeout>>`; all cleared in `onDispose`
- `bump()` / `subscribe` pub-sub pattern unchanged

### 3.6 Server plugin changes (`index.ts`)

- Reads `info.parentSessionID` in the event handler
- Passes `parentSessionID` and `agentLabel` into `JsonlRecord`
- Agent labelling logic duplicated from TUI (both processes are independent — no IPC)

### 3.7 `extractTokens` helper extension (`shared.ts`)

```typescript
// New field added to return value:
parentSessionID: info?.parentSessionID ?? null
```

### 3.8 Graceful degradation

If `parentSessionID` is never present in events (provider/version doesn't expose it), all sessions are labelled `"Main Agent"`. The sidebar shows one section per session, which is the same behaviour as today except with proper per-session sections instead of a single merged view.

---

## 5. Non-goals

- No cost calculation
- No charts or sparklines
- No configurable idle timeout (hardcoded 30s)
- No cross-session aggregation in the sidebar
- No persistence of agent tree across plugin restarts
